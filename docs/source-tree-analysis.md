### Test file locations (know before you add a test)

Tests are in **three** locations:

- `src/test/**` — integration and cross-cutting unit tests (most tests live here)
- `src/hooks/__tests__/**` — hook tests co-located with hooks
- `src/utils/__tests__/**` — utility tests co-located with utilities

This is inconsistent, but the split reflects historical patterns — match the neighbors of whatever you're testing. E2E specs live under `e2e/` and never mix with unit tests.

## `e2e/` — Playwright tests

```
e2e/
├── auth.setup.ts             # Setup project — signs in test user, stores state in .auth/user.json
├── auth.spec.ts              # Login, signup, password reset flows
├── classroom.spec.ts         # Create / rename / delete classroom
├── student.spec.ts           # Add / import / remove students
└── points.spec.ts            # Award / undo / class-wide award
```

Playwright is configured (`playwright.config.ts`) with:

- `fullyParallel: true` — every spec must be hermetic
- `storageState: '.auth/user.json'` — reused across tests to skip per-test login
- `reuseExistingServer: false` — a manually-started dev server could be pointing at prod Supabase, so Playwright always spawns fresh
- **Private-URL enforcement** — refuses to run unless `VITE_SUPABASE_URL` resolves to loopback/RFC1918/Tailscale CGNAT. This is a security boundary, not a lint rule.

## Entry Points & Bootstrap Order

1. `index.html` loads `/src/main.tsx` (Vite's default entry).
2. `src/main.tsx` calls `createRoot(document.getElementById('root')!).render(<StrictMode><App /></StrictMode>)`.
3. `src/App.tsx` mounts the context tree:
   1. `AuthProvider` — sets up Supabase session listener.
   2. `AuthGuard` — waits for session, renders `AuthPage` (login UI) until authenticated.
   3. `ThemeProvider` — applies persisted theme.
   4. `SoundProvider` — loads sound settings from `user_sound_settings`, preloads synthesized buffers.
   5. `AppProvider` — composes data hooks (`useClassrooms`, `useStudents`, `useBehaviors`, `useTransactions`).
4. `AppContent` (inside `AppProvider`) resolves the current view from `localStorage['app:view']`:
   - `'migration'` (takes precedence if `hasLocalStorageData()` returns true) → `MigrationWizard` (lazy)
   - `'home'` (default) → `TeacherDashboard` (lazy)
   - `'dashboard'` → `DashboardView` (lazy)
   - `'profile'` → `ProfileView` (lazy)
   - `'settings'` → `ClassSettingsView` (lazy)
5. `SyncStatus` mounts once at the root of `AppContent` (outside `<Suspense>`).

## Critical Folders — What Not to Restructure Casually

- **`src/contexts/`** — ordering and facade rule are load-bearing. Do not add contexts in parallel to `AppContext`; extend it instead unless the new state is auth-layer or below.
- **`src/hooks/`** — the one barrel `index.ts` is the intentional exception. Do not add a re-export barrel elsewhere.
- **`src/components/<feature>/`** — feature work goes into the existing folder. Do not invent parallel `features/` or `modules/` hierarchies.
- **`supabase/migrations/`** — migrations are append-only and sequential. Never edit a committed migration; add a new one.
- **`playwright.config.ts` private-URL guard** — parses the URL hostname rather than substring-matching. Don't weaken it.
