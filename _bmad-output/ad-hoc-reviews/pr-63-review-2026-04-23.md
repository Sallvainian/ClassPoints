# PR #63 Review — TanStack Query Phase 0+1

PR: <https://github.com/Sallvainian/ClassPoints/pull/63>

## Overview

Bootstraps `@tanstack/react-query` (QueryClient + centralized keys + DB→App transform) and migrates `useBehaviors` as the pilot. The legacy `useApp().behaviors` surface is preserved via an adapter in `AppContext`, so 45 consumer files don't move yet. Also adds a new TEA-style Playwright scaffold and hardens the E2E fail-closed host check.

The infrastructure work is solid and the scope discipline is good — this is a bootstrap PR that doesn't try to do Phase 2's optimistic mutations.

## Strengths

- **`queryKeys.ts` as single source of truth** — typed builders, consistent read/invalidate paths. PRD Risk 1 mitigation is real.
- **`structuralSharing: true` explicit, not default-inherited** — comment at `queryClient.ts:12-14` explains _why_ (ref-stable adapter output for the bridge). Load-bearing knowledge captured where it matters.
- **`playwright.config.ts` fail-closed host allowlist** — parses hostname via `new URL()` so `https://127.0.0.1.evil.com` can't slip through a substring match (line 40-46). Covers loopback + RFC1918 + Tailscale CGNAT. Good defense.
- **Direct `.env.test` parse instead of `loadEnv('', '')`** — the inline comment (lines 9-11) documents exactly why `loadEnv` was wrong (shell-env shadowing). Future-you will thank present-you.
- **Thin `useBehaviors`** — no manual loading/error state, sort helper is pure, mutations invalidate via shared key. Matches the spec.
- **Adapter bridge**: `behaviorsQuery.data ?? []` memoized (line 164), keeps ref identity when query returns stable data. Prevents consumer re-render storms.

## Issues & Risks

### 1. `ReactQueryDevtools` statically imported — DCE-dependent (`src/main.tsx:4,13`)

```tsx
import { ReactQueryDevtools } from '@tanstack/react-query-devtools';
...
{import.meta.env.DEV && <ReactQueryDevtools initialIsOpen={false} />}
```

NFR4 is satisfied today (`rg 'react-query-devtools' dist/` → 0), but this is fragile — any change that makes the devtools module side-effectful (top-level statement, barrel re-export change, `import type` transform drift) could silently start shipping ~200 KB to production. The documented contingency in ADR-4 is a DEV-gated dynamic import; worth adopting preemptively or at minimum asserting NFR4 in CI.

### 2. Naive `.env.test` parser misses quoted values (`playwright.config.ts:12-28`)

```ts
out[trimmed.slice(0, eq).trim()] = trimmed.slice(eq + 1).trim();
```

`KEY="value with spaces"` → `"value with spaces"` (quotes retained). `KEY=abc # comment` → `abc # comment`. Silent misconfig if a copy-pasted value has quotes. Use `dotenv.parse()` (transitively installed) or document the bare-values restriction in `.env.test.example`.

### 3. `resetBehaviorsToDefault` race (`AppContext.tsx:301-320`)

Acknowledged in `deferred-work.md`. One free win: `await refetchBehaviors()` on line 319 instead of fire-and-forget, so awaited callers see settled cache.

### 4. `loading` is now OR of four `isPending` values (`AppContext.tsx:181`)

v5's `isPending = no data AND no error` — so every fresh mount after gcTime eviction toggles the global `loading` flag. At `gcTime: 5min` + active use, probably a non-issue, but Phase 2 amplifies the surface. Consider `isLoading` (initial-fetch only) depending on UX intent.

### 5. Swallowed mutation errors return `null` (`AppContext.tsx:272-274`)

Legacy contract preservation, fine as intentional trade-off. Note consumers can't distinguish "null because no-op" from "null because failed". Phase 4 cleanup is the right call.

### 6. `dbToBehavior` assumes non-null `created_at` (`src/types/transforms.ts:15`)

`new Date(row.created_at).getTime()` returns `NaN` on null. DB presumably enforces non-null, but a narrow guard (`row.created_at ? ... : 0`) prevents silent NaN on any migration-path surprise.

### 7. Query keys scoped by nothing

`queryKeys.behaviors.all = ['behaviors']` — no user id. Single-tenant + RLS makes it safe, but a shared `QueryClient` across sign-in/sign-out transitions can flash user A's cache on user B's first render. Call `queryClient.clear()` on sign-out.

## Test Coverage

- Existing vitest unit tests intact; no new unit tests for the bridge itself (adapter identity, memoization). Reasonable — adapter dissolves at Phase 4, covered transitively by E2E.
- E2E scaffold in `tests/` is net-new, legacy `e2e.legacy/` preserved during transition. Good migration hygiene.
- No regression test for the realtime-removal race mentioned in `deferred-work.md`. Worth adding before Phase 2.

## Security

- `tests/support/helpers/supabase-admin.ts` throws without both `VITE_SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` — no silent fallback.
- Playwright host allowlist + `reuseExistingServer: false` eliminates the "shell env leaked prod creds" failure mode. Threat model documented inline.
- Confirm `.auth/user.json` is gitignored (likely under existing `.auth/` entry; didn't verify in diff).

## Verdict

**Approve with minor follow-ups.** Architecture sound, scope tight, test-infra security notably improved, known risks documented. Devtools static import and `await refetchBehaviors()` are the two items worth addressing before Phase 2 lands on top.

## Execution Protocol — HITL Safeguard (MANDATORY)

**Stop and ask the user before committing any code change** when implementation encounters ANY of the following. The agent MUST halt, surface the situation, and wait for explicit user approval before proceeding.

### Triggers that require stop-and-ask

1. **Conflict with a higher-order contract.** A prescription in this doc contradicts the PRD, architecture doc, existing ADRs, or a documented FR/NFR. Example: Patch #6 originally prescribed `refetchOnWindowFocus: false` globally, which silently violates PRD FR9 requiring `true` for non-realtime domains. That conflict required user input, not a judgment call.
2. **Ambiguity in prescription.** The spec can be read multiple ways and the "right" reading requires judgment (e.g., "dotenv.parse() vs dotenv.config() — which one?").
3. **Decision point not covered.** Something the spec didn't anticipate — e.g., "patch says fix the transform, but the transform has no callers."
4. **Scope expansion.** The fix requires touching more than the prescription named. Example: Patch #2 named one refetch site; grep revealed six. Fix all six or one?
5. **Implementation blocker.** Type errors, missing dependencies, test failures after the change, conflict with another unpushed commit.
6. **Unexpected discovery.** Schema says X but prescription assumed Y (e.g., Patch #3 dropped because `created_at` is NOT NULL — that discovery correctly paused implementation for user input).

### What NOT to do

- Make unilateral architectural decisions that silently amend the PRD, architecture doc, or prior ADRs
- "Advisor-check and commit" — advisor output informs the QUESTION to ask the user; it is not a substitute for user approval
- Assume silence is approval — if the user hasn't responded, the decision is not made
- Ship "defensible trade-offs" that regress a documented invariant (PRD FR/NFR, architectural contract) without explicit user sign-off
- Pick an option from a multi-option situation because it seems reasonable — the user picks, not the agent

### What stop-and-ask looks like

Halt before the commit. Post a message that includes:

- **Decision point encountered** (one sentence)
- **What the prescription literally says**
- **What the conflict / ambiguity / discovery is** (with evidence — file:line, PRD section, schema state)
- **2-3 options** with explicit consequences of each
- **Recommendation with reasoning**
- Wait for explicit user response. Do not proceed until they respond.

### Why this matters

Shipped work that violates user intent requires revert cycles that take longer than the original stop-and-ask would have. One lost hour from asking is strictly better than a three-hour revert + rework cycle on silently-shipped wrong decisions. HITL safeguard cost is bounded; silent-ship blast radius is not.

This rule applies during execution of this patch queue AND any future phase's patch queue. Future specs should reference this section by name.

1. **ADR-006 — user-scoped query keys.** Rewrite `queryKeys.behaviors.all` (and every subsequent domain) to take a `userId` parameter: `['behaviors', userId]`. Name the shared-iPad / account-switch-without-sign-out scenario explicitly as known-deferred.

## Gating Criterion for Future Phase Reviews

**"Any `useMutation` WITH `onMutate`"** — not "any `useMutation`." Invalidate-only mutations (`mutationFn` + `onSettled: invalidateQueries`) do not carry Phase 2 regression surface: no optimistic cache writes, no impure callbacks, no stale closures across re-renders. The surface opens when `onMutate` is introduced. Capture in ADR-005's Gate Criteria section so future phase-review PRs don't re-litigate this question from first principles.
